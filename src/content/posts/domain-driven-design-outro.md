## 01. 价值不在目录层级

DDD 的价值不在于增加目录层级，也不在于引入更多类、接口和设计名词。它真正要解决的，是复杂业务系统在长期演进中不断积累的几个问题：业务表达不清、模型边界混乱、技术细节侵入、系统耦合过高，以及团队沟通成本持续上升。

一个项目即使拥有 `application`、`domain` 和 `infrastructure` 目录，也不一定采用了 DDD；一个没有严格套用这些目录名称的项目，也可能拥有清晰的领域模型。

判断 DDD 是否产生价值，不能只看目录和类的数量。更有意义的标准，是代码能否比以前更准确地表达业务，以及重要规则是否集中在能够维护它们的模型中。与此同时，不同业务语境应当形成清晰边界，外部技术和外部模型的变化应得到有效隔离，开发者与领域专家也要建立可以持续演进的共同语言。

如果这些问题没有得到改善，再工整的包结构也只是建筑外立面。

## 02. 让代码成为业务知识的载体

传统过程式业务代码很容易演变为**事务脚本**：在一个 `Service` 方法中查询数据、判断状态、修改字段、调用外部接口、保存结果、发送消息，所有逻辑按执行顺序堆叠在一起。

这种代码短期看似直接，长期却很难维护。因为业务概念没有被显式建模，规则隐藏在流程和技术细节中。后来的维护者必须阅读完整调用路径，才能理解一个业务动作的真实含义。

例如，与其在应用服务中直接修改图书字段：

```java
book.setName(request.getName());
book.setPrice(request.getPrice());
book.setUpdatedAt(LocalDateTime.now());
```

不如让领域对象表达自己的行为：

```java
book.renameTo(command.name());
book.changePriceTo(command.price());
```

后者的价值不只是命名更加优雅。`renameTo` 和 `changePriceTo` 为业务动作建立了唯一入口，相关校验也可以在模型内部得到一致执行。

统一语言进一步让代码成为业务知识的载体。如果业务人员说的是“下架图书”，代码里却叫 `deleteBook`；业务人员说的是“冻结账户”，代码里却叫 `updateStatus(3)`；业务人员说的是“订单履约”，代码里却分散着多个无语义的方法，那么代码和业务之间就会产生持续的翻译损耗。

下面这些方法名本身就在描述业务事实：

```java
order.cancelBecausePaymentExpired();
book.markAsSoldOut();
account.freezeDueToRiskControl();
```

类名、方法名、事件名和异常名越接近团队真实使用的业务语言，开发者就越容易与领域专家讨论模型，代码评审也越容易发现业务理解上的偏差。

这并不意味着方法名越长越“领域化”，也不意味着所有内部细节都要采用业务术语。统一语言的目标是减少关键概念的翻译损耗，而不是排斥必要的技术词汇。

## 03. 用边界控制复杂度

软件复杂度通常不是一次性产生的，而是在持续迭代中逐渐扩散。一个对象最初可能只服务于简单场景，后来被越来越多模块复用，字段越来越多，方法越来越杂，最终成为无人敢改的核心类。

限界上下文和聚合边界的价值，就在于阻止这种复杂度无限蔓延。限界上下文首先承认不同语境下的模型不必强行统一，让每套模型只承担自己的语义；聚合进一步约束对象之间的关系，避免所有对象都直接连接。当不同模型确实需要协作时，上下文映射则用来明确它们之间的依赖和协作方式。

边界不是为了制造孤岛，而是为了让变化产生可预期的影响。一个模型只要能在自己的上下文中保持一致，就不必承担其他上下文的全部语义。

以“用户”为例，认证上下文可以拥有 `Account`，订单上下文可以拥有 `Buyer`，营销上下文可以拥有 `Member`，风控上下文可以拥有 `RiskSubject`。这些模型可以通过 ID、事件或防腐层关联，却不必共享一个巨大的 `User` 对象。

当认证规则改变时，订单上下文不应该被迫调整无关字段；当会员等级变化时，也不应该要求所有依赖“用户”的模块同时发布。清晰边界把一次全局变更变成局部演进。

当然，边界本身也需要成本。跨上下文查询、事务和协作会比单体对象图更加显式。DDD 的选择不是在“复杂”与“简单”之间二选一，而是在业务本来就存在复杂性时，决定把复杂性隐藏在偶然耦合中，还是通过模型和边界显式管理。

## 04. 隔离技术细节并提高可测试性

许多系统的领域代码会逐渐被技术细节污染。这种污染往往始于一些看似方便的妥协：实体中开始充满 ORM 映射和数据库字段设计的痕迹，业务逻辑逐渐依赖 Web 框架的请求对象，领域对象甚至直接调用第三方 API。随着类似做法不断累积，缓存、事务、消息和业务判断被一起塞进 `Service` 方法，数据库表结构也开始反向决定对象模型。

这些问题会让系统越来越“技术中心化”。业务规则被迫迁就具体实现，模型也难以独立演进。

DDD 通过资源库、端口、适配器和防腐层隔离技术细节。领域模型不需要知道数据来自 PostgreSQL、Elasticsearch 还是远程服务，也不需要知道事件最终发送到 RabbitMQ、Kafka 还是进程内事件总线。

这种隔离的意义并不是所有技术都能毫无成本地替换，而是让更换技术时，变化尽量停留在与技术有关的边界内。业务知识通常比框架版本和基础设施选择更加稳定，理应获得更强的保护。

隔离技术也会提升业务规则的可测试性。如果规则散落在 `Controller`、`Service`、`Repository` 和远程调用之间，测试往往需要启动 Spring 容器、准备数据库并模拟多个依赖，反馈缓慢且失败原因不清晰。

当核心规则被封装在实体、值对象、聚合和领域服务中时，许多测试可以脱离框架直接执行：

```java
@Test
void should_not_allow_negative_price() {
    assertThrows(
            IllegalArgumentException.class,
            () -> Price.of(
                    Currency.getInstance("CNY"),
                    new BigDecimal("-1")
            )
    );
}
```

这类测试聚焦业务规则本身，不依赖 Web 框架、数据库或消息队列。它们更快、更稳定，也更能在重构时保护核心模型。

不过，可测试性并不只来自“用了 DDD”。如果领域对象依然依赖静态环境、系统时间和隐藏的全局状态，即使它们位于 `domain` 目录中也很难测试。真正发挥作用的，是清晰职责、显式依赖和可控边界。

## 05. DDD 与微服务

DDD 与微服务经常被放在一起讨论，因为二者都强调边界，但它们关注的层面不同。

微服务是一种架构风格，关注服务如何拆分、部署、通信和治理；DDD 是一种建模方法，关注业务边界如何识别、模型如何表达、规则如何组织。

DDD 不要求微服务。一个模块化单体同样可以拥有多个限界上下文、独立模型和清晰契约。反过来，把系统拆成许多可独立部署的进程，也不代表业务边界已经合理。

如果缺少领域建模，微服务拆分很容易退化为按照技术层、数据库表或团队分工进行机械切割。这样的服务虽然在部署上分开，业务上却仍然高度耦合，最终形成“分布式单体”。

一个常见的反例，是把所有场景需要的信息都放进所谓的“用户服务”。订单服务为了完成业务而同步查询大量用户字段，营销、会员、权限和风控服务也逐渐依赖同一个用户模型。久而久之，任意一个字段变化都会影响多个服务，一次业务操作也必须同步调用一长串服务才能完成。

表面上看，这是多个微服务；实质上，它们共享同一个隐含的全局模型，也共同承担一条脆弱的分布式事务链。

DDD 可以为服务拆分提供业务依据。限界上下文是值得优先考虑的服务边界候选，而数据库表或 CRUD 模块通常不是。

但限界上下文也不必自动对应一个微服务。服务拆分还要考虑团队规模、部署频率、性能、可用性和运维成本。在这些因素没有提出独立部署需求前，把多个上下文保留在一个模块化单体中，往往更加经济。

## 06. 服务自治与演进

微服务的核心目标之一是服务自治。一个服务如果拥有清晰的领域边界、独立的数据模型和稳定的外部契约，就可以在较小范围内独立演进。

DDD 为这种自治提供了建模基础。限界上下文先划定服务的语义范围，聚合再把事务一致性约束在明确边界内。在此基础上，领域事件负责表达上下文内部发生的业务事实，应用层可以将需要对外传播的事实转换为稳定的集成事件，以支持跨上下文协作。防腐层负责隔离外部模型的变化，上下文映射则帮助团队看清彼此之间的依赖和协作关系。

例如，在订单创建后，订单上下文可以产生 `OrderCreated` 领域事件。应用层将它转换为稳定的集成事件后，库存上下文可以预留库存，积分上下文可以累计积分，通知上下文可以发送消息。

订单上下文不需要理解这些下游上下文的内部规则，也不必因为增加了新的订阅者而修改自己的核心模型。每个上下文只处理自己负责的业务事实。

这种事件驱动协作并不意味着所有场景都应该异步化。用户必须立即得到结果的操作、需要强一致性的聚合内部规则，以及失败后难以补偿的流程，仍然需要审慎设计同步边界。

服务自治也不是“互不往来”。真正的自治意味着通过明确契约协作，并能够在契约允许的范围内独立改变。没有契约的自由调用不是自治，而是把内部耦合搬到了网络上。

## 07. 成本与适用边界

DDD 的价值明显，但它并不适用于所有系统。

采用 DDD 意味着持续建模，而不是搭好一套目录结构后便一劳永逸。团队需要在与领域专家的反复沟通中提炼统一语言，再以此识别子域、划定限界上下文并梳理上下文之间的关系。业务发生变化后，模型也要随认识更新；如果旧结构已经无法准确表达业务，就应及时重构，而不是不断叠加条件分支。

边界越清晰，系统中的转换和协作代码通常也越显式。这些额外代码，以及持续沟通和维护模型所投入的时间，都是采用 DDD 必须承担的成本。只有当业务本身足够复杂时，这些成本才可能换来更清楚的表达和更可控的变化。

如果系统只是简单的 CRUD，业务规则很少、生命周期较短，需求变化也不复杂，那么完整采用 DDD 很可能只会增加不必要的抽象。此时，清晰的分层、准确的命名、良好的模块化和必要的自动化测试通常已经足够。

反过来，当业务规则复杂到难以用流程代码准确表达，或者系统需要长期承载不断演进的业务知识时，DDD 才更容易发挥价值。需求频繁变化、多团队共同开发，会进一步放大统一语言和明确边界的重要性；如果模块职责已经混乱、全局模型持续膨胀，或者团队正在从业务能力的角度评估微服务边界，领域建模也能提供更可靠的拆分依据。与此同时，端口和适配器等边界还能保护核心业务，避免基础设施变化频繁向内渗透。

团队也不必一次性采用 DDD 的全部模式。可以先从最有价值的核心子域开始，建立统一语言和清晰模块；随着规则复杂度上升，再引入值对象、聚合、领域事件和更严格的端口边界。

这种渐进方式比在整个系统中批量创建 `Entity`、`Repository` 和 `DomainService` 更符合 DDD 的初衷：把建模成本投入真正复杂且重要的问题。

## 08. 小结

归根结底，DDD 先让代码回到业务本身。统一语言拉近了开发者与领域专家的距离，也让代码能够直接承载业务知识；限界上下文控制模型语义的范围，聚合则在范围之内维护一致性，避免对象关系无序扩张。

在模型之外，端口、资源库、适配器和防腐层共同隔离技术与外部模型的变化。与微服务结合时，这些边界还能为服务拆分和自治提供业务依据。清晰的职责与显式的依赖最终汇集到同一个结果：核心规则更容易测试，系统也更容易随业务认识一起演进。

领域驱动设计不是一套可以机械套用的模板，而是一种面对复杂业务的软件设计思维。它要求开发者不只关注技术实现，还要理解业务语义；不只追求代码可以运行，还要追求模型能够表达、边界可以维护、系统能够演进。

在简单系统中，DDD 可能显得繁复；但在复杂系统中，它往往是抵抗混乱、延缓腐化和沉淀业务知识的重要方法。
